# Wizard Code Review — Findings & Action Plan (2026-07-11)

Audience: coding agent picking up remediation work on this repo.
Scope reviewed: `packages/core`, `packages/cli`, `packages/schemas`, docs, at commit `253edbc` plus the uncommitted working-tree changes on `main`.

## Overall verdict

The architecture is sound and should be kept: local/remote boundary, versioned Zod contracts, plan → preview → approve → apply separation, privacy-by-default. The problem is inverted investment — the safety machinery (digest binding, atomic writes, rollback, path hardening) is more mature than the product core. **A fully approved, applied, and "verified" setup currently produces a dead integration that sends zero events.** Fix the happy path and the flow friction before adding more rigor or more verification layers.

Work priority order:

1. P0 — make the deterministic install actually work end-to-end
2. P0 — fix the two security-claim gaps (approval forgery, verify false-pass)
3. P1 — collapse flow friction (artifact-by-reference, error taxonomy, approval bridge)
4. P2 — smaller correctness/robustness items
5. P2 — test coverage that would have caught the above

## P0-A: The deterministic happy path does not produce a working install

### 1. Next.js `src/app` / `src/pages` layouts not detected — HIGH

- `packages/core/src/inspector.ts:319-337` — `detectFramework` only checks `join(root, "app")` and `join(root, "pages")`. The `create-next-app --src-dir` layout (very common) falls through to `framework: "node"` (confidence 0.75).
- Cascade: `environmentDefaults` (`packages/core/src/setup-plan.ts:152-168`) picks `USERMAVEN_PUBLIC_KEY` instead of `NEXT_PUBLIC_*` → the key is never exposed to the browser bundle → the generated client silently initializes as null → zero tracking, no error. Target path also becomes `src/usermaven.ts` without `"use client"`.
- Fix: also check `src/app` and `src/pages`; add a `next-src-app-router` fixture (or reuse existing framework enum values with the src prefix handled in path selection).

### 2. Generated client is never wired into the application — HIGH

- `packages/core/src/setup-plan.ts:380-389` — the plan creates `app/usermaven-client.ts` / `src/usermaven.ts` with `autocapture: false, autoPageview: false`, but emits **no operation** importing it from `layout.tsx` / `_app.tsx` / `main.tsx` and no pageview/route-change tracking. Unless the AI instrumentation proposal happens to add the import, the applied setup emits nothing — not even pageviews.
- Fix: emit a deterministic wiring operation (entry-point import + pageview tracking) per framework, so an approved+applied plan works without depending on the AI proposal.

### 3. Generated client invalid for `react` / `node` / `unknown` frameworks — HIGH

- `packages/core/src/setup-plan.ts:183-228` — `environmentExpression` emits `process.env.USERMAVEN_PUBLIC_KEY` in a browser client module. CRA needs the `REACT_APP_` prefix; webpack 5 / esbuild / plain browsers don't shim `process` → `ReferenceError: process is not defined` at module load, or a silent null client where the bundler defines `process.env` as `{}`.
- Static verification does not catch this: `public-config-references` (`packages/core/src/verify.ts:378-402`) only greps for the env-var _name_.
- Fix: emit framework-correct env access per bundler, or refuse unsupported frameworks with a clear risk instead of emitting broken code.

## P0-B: Security claims that don't hold

### 4. Approvals are fully forgeable at the MCP boundary — HIGH

- `packages/cli/src/mcp-server.ts:346-366` — `apply_changes` accepts the entire `ChangeApproval` object inline (`approval: changeApprovalSchema`). Every field is derivable without user interaction: `plan_digest` = sha256 of the canonicalized plan the agent already holds; `repository_root_fingerprint` = sha256 of the realpath string (`packages/core/src/approval.ts:32-36`); `approved_by` is a fixed literal; timestamps freely chosen. No nonce, HMAC, signature, or server-side registry. The one-time record only prevents replay of a given `approval_id`, not minting a fresh one.
- This defeats `docs/threat-model.md:40` ("Forged, broadened, or replayed approval → Mint approval only after exact interactive confirmation") and `docs/apply-playbook.md:49-50`.
- Fix (preferred): `apply_changes` accepts only an `approval_id` and loads the artifact the interactive CLI wrote under `.usermaven/` (mode 0600); optionally HMAC with a per-checkout secret the MCP client never sees. Alternative minimum: downgrade the threat-model and tool-description claims to match procedural reality.

### 5. Verification false-pass when env-var names are absent — HIGH

- `packages/core/src/verify.ts:355-360, 378-387` — non-null assertions on optional fields (`plan.workspace.key_env_var!`). Schema optionality: `packages/schemas/src/setup-plan.ts:92-101`. When absent, `String.includes(undefined)` coerces to the literal string `"undefined"`, which nearly every JS/TS file contains → `public-config-references` passes without checking anything. `scanPublicConfigReferences` has the same coercion.
- Fix: skip or restructure the check when the fields are unset; add a test with an env-var-less plan.

### 6. Security digest is locale-dependent and copy-pasted — MED

- `canonicalize` sorts keys with `localeCompare` in three copies: `packages/core/src/approval.ts:16`, `packages/core/src/workflow.ts:81`, `packages/core/src/tracking-plan.ts:22`. Ordering depends on process locale/ICU build; the CLI (approve) and MCP server (apply) are separate processes, so differing `LANG` can produce spurious "Approval does not match the exact setup plan" failures.
- Fix: one shared `canonicalize` using code-point comparison (`a < b ? -1 : 1`); add a cross-locale digest-stability test.

### 7. Root fingerprint and `run_check` are not content-bound (approve→apply TOCTOU) — MED

- Root fingerprint = sha256(realpath) only; directory contents can be swapped between approve and apply, and approvals are portable across machines with identical paths. `run_check` (`packages/core/src/apply.ts:452-453`) executes whatever `package.json`'s `build` script says _at apply time_ — unpinned across the approval window.
- Fix: bind dev/inode or `.git` HEAD digest into the fingerprint; hash the `scripts` block into `run_check` operations.

## P1: Flow friction (this is where the product wins or loses)

Current state: 8 commands passing ~7 hand-managed JSON artifacts. Humans can't complete it (`--ai-proposal` / `--ai-instrumentation` are documents only a model can author). Agents can't complete it (error masking + approval-gate dead end, below).

### 8. MCP error masking destroys agent self-correction — HIGH

- `packages/cli/src/mcp-server.ts:125-139` — every non-`ScopedPathError` collapses to `{code: "operation_failed", message: "Local operation failed without returning repository contents"}`. Core produces rich, non-sensitive failure reasons (stale file hash, expired/replayed approval, digest mismatch, uncovered tracking item); agents get none of them and can only blind-retry.
- `packages/schemas/src/agent-event.ts:52-59` already defines the needed `error {code, message, retryable}` shape — nothing ever emits it (dead exported surface).
- Fix: map core failures to a sanitized error-code taxonomy (`approval_expired`, `approval_replayed`, `stale_file_hash` + relative path, `coverage_missing` + item key, `plan_digest_mismatch`, `validation_failed` + field paths). None leak repository contents.

### 9. Approval gate strands the agent with no structured hand-off — HIGH

- `packages/cli/src/mcp-server.ts:345-357` + `packages/cli/src/cli.ts:235-237` (approve requires TTY on stdin _and_ stdout). The agent must write the plan to a file itself, guess the exact `usermaven-wizard approve <plan> --operations <ids> --root <same-realpath> --output <file>` invocation, get the user to run it within the 15-min TTL (`cli.ts:247`), and replay the file into `apply_changes`. Any deviation fails with the masked error from finding 8.
- Fix: when `apply_changes` lacks a valid approval, return a structured `approval_required` result (shape already in `agent-event.ts:30-37`) containing the plan file path, operation IDs, and the exact approve command to relay to the user; repeat the recipe in the tool description.

### 10. No artifact-by-reference; full plan round-trips through model context 4+ times — MED

- `packages/cli/src/mcp-server.ts:328, 351-356, 384-388, 413-418` — four tools take the entire `setupPlanSchema` (embedding full generated-file contents) inline; results duplicate the payload in `content[0].text` and `structuredContent`. `tools/list` inlines several deeply nested zod→JSON schemas — real hosts (Cursor etc.) truncate or reject large tool schemas.
- The checkpoint system already stores artifact paths + digests under `.usermaven/workflows`.
- Fix: auto-checkpoint from every producing command; let `preview_changes` / `apply_changes` / `prepare_verification` / `verify_setup` (and CLI equivalents) accept `workflow_id` + `plan_digest` instead of the inline plan. This also gives `approve` a canonical on-disk plan file.

### 11. No guided human flow — MED

- `inspect` output is consumed by nothing (`cli.ts:164` and `mcp-server.ts:272` both re-inspect internally). No `wizard setup` orchestrator, no default artifact locations, no stdin/`-` support; `checkpoint`/`resume` are manual ceremony (`cli.ts:351-378`).
- Fix: add an interactive `wizard setup` orchestrator with default artifact locations under `.usermaven/`, and a `wizard next` wrapping `resume` that prints the exact next invocation.

### 12. Agent busywork the pipeline should absorb — MED

- Agents must compute sha256 `before_hash` values and hand-author unified diffs with no affordances; `inspect_project` returns paths but no file digests → include per-file sha256 digests and entry-point hints in `ProjectInspection`.
- Verification requires hand-assembling `verification-evidence.json` from a template plus a 4-step manual fingerprint procedure → auto-create the verification session at successful apply; merge runtime+transport evidence into one observer-emitted artifact.
- Env setup is a fully manual step even though writing `KEY_NAME=` placeholder lines to `.env.example` is mechanical (currently blocked by the protected-path rule with no assisted alternative).

## P2: Test coverage gaps that would have caught the above

- No Next.js `src/app` fixture (finding 1). Fixtures cover only the three happy layouts.
- Nothing compiles or executes the generated client — only string-contains assertions (findings 2, 3).
- No verify test with a plan lacking env vars (finding 5).
- No `approval.test.ts` at all: TTL bounds, manual-step-only rejection, unknown/duplicate op IDs, cross-root fingerprint mismatch untested (finding 4 adjacents).
- No successful `edit_file` path through `generateSetupPlan` — only rejection paths — so the diff-regex false rejection (13) and the happy path are untested.
- No cap-overflow tests (15); no non-npm package-manager apply tests (18); no monorepo/lockfile-above-root inspector case (17); no corrupt-record resume (23); no cross-locale digest test (6); no concurrent-apply race; workflow forward-jumps with self-supplied artifacts are accepted and untested.

## Suggested execution order for remediation

1. Findings 1–3 together (framework detection + env conventions + wiring op) — one coherent change with new fixtures; this makes the product work.
2. Findings 5 and 6 (verify false-pass + shared code-point canonicalize) — small, high-value correctness fixes with tests.
3. Finding 4 (approval-by-id at the MCP surface) — security architecture change; update threat model + apply playbook in the same PR.
4. Findings 8 + 9 (error taxonomy + approval_required bridge) — unlocks agent flows; wire the existing `agent-event.ts` shapes.
5. Finding 10 + 11 (checkpoint-backed artifact references + `wizard setup` / `wizard next`) — flow collapse.
6. P2 table + test gaps opportunistically, batched by file.

Verification for each phase: `npm run check`, then `node packages/cli/dist/cli.js inspect fixtures/<fixture>` end-to-end against fixtures, plus `scripts/mcp-stdio-smoke.mjs` for MCP changes.
